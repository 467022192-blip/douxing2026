# 攻略页体验优化设计

## 背景

当前 `AITripPlanner` 页面已经具备“输入需求 -> 生成 3 套景点方案 -> 保存历史”的基础能力，但存在以下明显问题：

- 线上生成耗时偏长，复杂请求常达到 50 秒以上，用户容易误判为卡死。
- 页面文案偏工程化，例如“把需求说清楚，给你 3 套行程灵感”，产品感不足。
- 首屏信息结构偏“功能页”，与目标中的“攻略助手”体验不一致。
- 底部导航中的 `AI规划` 文案和图标辨识度一般，不足以表达 AI 驱动属性。

本次优化目标是在不推翻现有数据结构和接口的前提下，把 `AI规划` 升级为更偏产品化的“攻略”体验。

## 用户确认的方向

- 页面名称从 `AI规划` 统一改为 `攻略`。
- 视觉上只借鉴参考图的核心思路，不做高相似复刻。
- 精简首屏非必要说明文字，减少“开发工程表达”。
- 页面保持单页结构，不新增顶部双入口。
- 底部导航中将 `攻略` 调整到中间第 3 个 tab。

## 目标

- 降低 AI 生成的平均等待时间和长尾等待时间。
- 在无法显著缩短所有请求耗时的前提下，提升等待过程的可感知性与可控性。
- 将页面从“工具页”升级为“攻略助手页”。
- 保留当前 demo 提问方式与结果能力，避免功能倒退。

## 指标与验收

以当前线上复杂请求常见 `50s+` 为已知背景，本次优化目标定义为：

- `p50` 生成总耗时尽量压到 `20s` 以内。
- `p90` 生成总耗时尽量压到 `35s` 以内。
- `50s+` 的长尾请求比例要明显下降。
- 用户在生成开始后的 `1s` 内必须看到明确等待反馈。
- 慢请求场景下，用户不能只看到无限旋转，必须看到阶段提示或兜底操作。
- 因结构不完整导致的服务端失败率不能高于当前基线。

首轮验收以以下观察口径为准：

- 总耗时：前端点击生成到结果渲染完成。
- 模型耗时：服务端调用模型开始到模型返回。
- 匹配耗时：景区数据准备与匹配所花费时间。
- 成功率：成功返回 3 套结构完整攻略的比例。
- 慢请求占比：超过软阈值和长阈值的请求比例。
- 重试率：用户点击重新生成的比例。

## 非目标

- 本次不实现语音输入。
- 本次不改为流式输出协议。
- 本次不引入酒店、餐厅等新推荐类型。
- 本次不重做保存历史的数据结构。

## 方案概览

采用“前后端一起优化，但不推翻接口形态”的方案：

- 后端降低模型生成成本与景区匹配成本。
- 前端增加阶段式等待提示、超时兜底与更产品化的输入/结果呈现。
- 导航与页面文案统一切换为“攻略”。

## 信息架构

### 底部导航

底部导航顺序调整为：

1. 首页
2. 行程
3. 攻略
4. 动态
5. 我的

其中 `攻略` 位于中间位置，并使用带 AI 感的异形入口图标。

### 攻略页结构

首屏采用更聚焦的单列结构：

1. 轻量顶部区域
2. AI 主视觉 + 简短标题
3. 核心输入卡片
4. 生成中的提示或攻略结果
5. 历史攻略区

### 首屏文案

- 主标题：`我可以帮你做攻略`
- 副标题：`说下天数、出发地和偏好，帮你生成 3 套景点攻略`
- 输入 placeholder 保留示例：
  `5天4夜的行程，一家三口，有个6岁的小孩，想去海边，从北京出发`

不再保留偏工程式文案：

- `把需求说清楚，给你 3 套行程灵感`
- `首版先支持文字`
- `推荐结果`

## 视觉设计

### 总体风格

- 保持项目当前移动端 H5、简洁、轻量的风格。
- 颜色以浅色背景和柔和蓝绿渐变为主，避免大面积深色或过重 Emerald 实底。
- 减少“功能面板感”，增加“内容服务感”。

### AI 主视觉

- 在首屏输入框上方增加简洁 AI 头像或抽象球体插画。
- 右上角叠加 `AI` 小徽标，强化“AI 驱动”感。
- 不引入复杂插画系统，保持轻量。

### 输入卡片

- 输入区改为高权重白色大卡片。
- 使用浅蓝/浅紫边框或阴影做区分，贴近参考图的“被重点托起”的感觉。
- 主按钮采用更明显的发送感或生成感图标。
- 示例问题收敛为 1 到 2 个精选示例，不保留过多胶囊按钮。

### 结果卡片

- 将“方案卡”改为更像“攻略卡”的内容样式。
- 弱化“方案 1/2/3”的机械编号展示，可保留但更轻。
- 减少冗长 reason 展示，控制摘要长度。
- 每日景点块更易扫读，优先展示标题、景点数和关键景点。

## 性能优化设计

### 模型输出减负

对 `api/ai-trip-plans.ts` 做以下优化：

- 精简 prompt 中的说明文字，避免重复约束。
- 缩短 JSON 示例，减少无效 token 消耗。
- 下调 `max_tokens`，优先控制在满足 3 套攻略结构的最小合理区间。
- 适度下调 `temperature`，减少模型发散造成的生成时长波动。

同时补充质量护栏，避免只降耗时但提升失败率：

- 服务端对模型返回继续保留结构校验。
- 结构校验失败时，自动做 1 次轻量重试。
- 若仍失败，返回更明确的产品化错误文案，而不是技术错误直出。
- 当模型返回内容接近可用时，可做基础 JSON 清洗与代码块剥离，不直接失败。

### 景区匹配减负

当前每次请求都会拉取最多 3000 条景区数据后再逐条匹配，需要优化：

- 在 serverless 运行期内缓存景区基础数据。
- 缓存仅保留匹配必要字段，如 `id/name/province/city/address`。
- 尽量复用已加载的景区数据，减少每次调用 Supabase。
- 为缓存设置明确 TTL，例如 `10 分钟`。
- 缓存未命中或已失效时，自动回退到实时查询。
- 增加缓存命中和未命中观测，便于判断后续是否需要更重的索引方案。

如果首轮优化后耗时仍高，再评估：

- 构建标准化名称索引。
- 缩小需要参与匹配的景区范围。
- 持久化缓存或预计算索引。

### 前端请求控制

- 生成中禁用重复点击。
- 对连续相同输入的重复请求做短时间保护。
- 保留“重新生成”，但避免瞬时重复触发多次网络请求。
- 使用 `AbortController` 管理请求生命周期。
- 为每次生成分配本地 `requestId`，仅允许最新请求回写页面结果。
- 新请求开始时，应取消旧请求或让旧请求结果失效，避免旧结果覆盖新结果。

## 等待态与超时兜底

### 请求状态机

前端生成流程按以下状态机实现：

`idle -> generating -> soft-timeout -> done / failed`

需要重新生成时：

`generating / soft-timeout -> retrying -> generating`

状态约束如下：

- `idle`：可编辑输入，可点击生成。
- `generating`：主按钮禁用，展示阶段提示，可允许用户查看页面其他内容。
- `soft-timeout`：保留当前请求，展示“继续等待 / 重新生成”。
- `retrying`：若用户选择重新生成，先取消旧请求，再发起新请求。
- `done`：仅最新 `requestId` 可落结果。
- `failed`：展示错误文案与重试入口。

“继续等待”的定义：

- 不发新请求。
- 保留当前请求继续等待。
- 继续展示更柔和的慢请求提示。

“重新生成”的定义：

- 取消当前请求。
- 清空本轮等待态。
- 以新的 `requestId` 发起请求。

### 分阶段提示

页面不依赖流式协议，而是基于本地时间切换 3 段提示：

1. `正在理解你的需求`
2. `正在组合景点攻略`
3. `正在整理 3 套推荐`

建议阈值：

- `0-6s`：正在理解你的需求
- `6-15s`：正在组合景点攻略
- `15s+`：正在整理 3 套推荐

### 预计等待时间

生成中在按钮或输入卡片下方展示：

- `通常 15-30 秒，复杂需求会更久`

### 慢请求安抚

超过短阈值后展示：

- `这次需求更复杂，正在继续生成`

建议软阈值：`20s`

### 超时兜底

超过长阈值后，展示明确操作而不是无限旋转：

- 继续等待
- 重新生成

- `如果你刚刚描述得很详细，生成时间会更长一些`

建议长阈值：`35s`

交互约束：

- 进入长阈值后，输入框允许继续编辑，但不会影响当前请求。
- 用户若修改输入后点“重新生成”，必须以新内容发起新请求。
- 若请求被取消，不展示被取消请求的最终返回结果。

## 历史攻略区

- 继续保留历史攻略功能。
- 视觉层级下沉，避免和输入区争夺首屏注意力。
- 区域标题统一为 `我的攻略`。
- 卡片文案精简，保留输入摘要、生成时间和攻略数量。

## 图标与命名规范

### 页面命名

- 路由保持 `/ai-trip-planner` 不变，降低改动风险。
- 对外展示文案统一使用 `攻略`。

### 图标方案

- 底部导航使用自定义 AI 感图标替代默认 `Sparkles`。
- 图标形态采用“主体图标 + AI 小角标”的组合方式。
- 风格保持简洁、线条轻，不使用厚重描边。
- 图标需兼容 `320 / 375 / 390` 宽度移动端。
- 选中态与未选中态都需清晰，且文案不折行。

## 组件边界

建议把当前页面拆出以下可复用模块，降低 `src/pages/AITripPlanner/index.tsx` 的复杂度：

- `GuideHero`
- `GuideComposer`
- `GuideGeneratingState`
- `GuideResultCard`
- `GuideHistoryList`
- `GuideTabIcon`

## 错误处理

- 保留当前服务端错误提示能力。
- 用户侧错误文案改得更产品化，减少技术术语直出。
- 若 AI 请求失败，优先展示：
  - 发生了什么
  - 可以怎么做，例如重试
- 请求取消时不提示为失败，只静默进入新的生成流程。
- 结构校验失败时，优先提示“这次生成不够稳定，请重新试一次”。

## 测试与验证

至少验证以下路径：

- 输入示例需求并成功生成 3 套攻略。
- 生成过程能看到阶段提示与预计等待时间。
- 慢请求时能看到慢请求提示。
- 超时后能看到“继续等待 / 重新生成”兜底交互。
- 保存攻略后，历史区能正确回显。
- 底部导航顺序变更正确，`攻略` 位于第 3 个。
- 新图标和文案在移动端宽度下不折行、不挤压。
- 被取消的旧请求不会覆盖新请求结果。
- 软超时和长阈值状态切换符合预期。

## 风险

- 调低 `max_tokens` 后，可能导致个别复杂需求下返回结构不完整。
- serverless 运行期缓存对冷启动无效，只能优化热实例表现。
- 页面视觉调整较大，需注意不要破坏现有移动端布局稳定性。

## 回滚原则

若出现以下任一情况，应优先回退参数或功能细节：

- 结构失败率明显上升。
- 超时比例没有改善但错误率升高。
- 攻略页主链路生成成功率下降。
- 新导航图标或顺序导致明显误触或布局异常。

## 推荐实施顺序

1. 优化 API 性能参数与景区缓存
2. 改底部导航顺序、命名与图标
3. 重做攻略页首屏结构与输入卡片
4. 增加等待态、分阶段提示和超时兜底
5. 优化结果卡片与历史区
6. 完成构建与手动验证
